Klassen-Variable und Konstante

Für einige Werte, die wir bisher verwendet haben, ist es sinnvoll, ihre Definition zu ändern. Da wir bisher Werte nur in Instanz-Variablen speichern konnten, haben wir das auch in jedem Fall gemacht. 

Klassen-Variable verwenden

In einigen Fällen ist es allerdings sinnvoll, statt dessen diese Werte in Klassen-Variablen zu speichern. Wann ist das allgemein der Fall?

Antwort: Wenn der Wert in allen Instanzen gleich ist. Dann verbraucht das Speichern in jeder einzelnen Instanz nur unnötig Speicherplatz. Außerdem ist dann die Logik, dass der Wert für alle Objekte gilt damit nicht im Code sichtbar.

Konstanten verwenden

Es gibt einige Werte, die sich über die Laufzeit des Programms nicht ändern sollen. Der Wert ist also eine Konstante.

Warum sollen Konstante als Klassen-Variable definiert werden?

Antwort: Weil sich ihr Wert nicht ändern soll, ist er auch für alle Instanzen gleich. Der Wert sollte also für alle Instanzen nur einmal gespeichert werden.  

Konstante definieren

Werte, die im Spiel diesen Kriterien entsprechen sind:

  • die Angabe der Zeilenhöhe und Spaltenbreite
  • der Text, der als Reaktion des Ende-Buttons verwendet wird
  • das Array mit den Strings der erlaubten Farben 
    • Nur, falls Sie ein solches Array bereits verwendet haben.

Definieren Sie diese Werte als Konstanten in den passenden Klassen. Denken Sie dabei an die Konvention zur Namensgebung.

Der Reaktions-Text muss in mehreren Klassen verwendet werden können.

Konstante aufrufen

Da sie nun sowohl den Namen als auch die Art der oben modifizierten Werte geändert haben, müssen Sie nun auch ihre Aufrufe anpassen.

Passen Siedie Aufrufe entsprechend an. Achten Sie darauf, dass Sie jeweils den Namen der Klasse angeben. Das ist zwar nicht immer nötig, macht den Code aber klarer, besser verständlich und wartbarer.

Tipp: um alle zu ändernden Aufrufe zu finden, können Sie für die Zeilenhöhe und Spaltenbreite den Compiler suchen lassen.

Laufende Nummer für die ID einer Karte

Beim Erzeugen eines Karten-Objekts wird auch eine eindeutige ID erzeugt. Bisher hatten wir diese aus den Parametern Zeile und Spalte zusammengebaut. Das funktioniert aber nur in diesem Spezialfall. Häufig wird für eine eindeutige ID eine laufende Nummer hochgezählt.

So eine laufende Nummer kann nicht in einem einzelnen Karten-Objekt gespeichert und verwaltet werden. Ihr Wert muss getrennt von den Instanzen der Klasse Karte gespeichert und hochgezählt werden. Das ist ein typischer Fall für eine Klassen-Variable: Es gibt sie 1x für alle Instanzen der Klasse. 

Laufende Nummer definieren

Definieren Sie daher eine solche laufende Nummer als Klassen-Variable. Ihr Wert soll bei 42 beginnen. 

Laufende Nummer verwenden

Für die Erzeugung des Strings für die ID der Karte soll nun das Literal "ID" mit der oben definierte Laufende Nummer konkateniert werden. Danach soll die Laufende Nummer um 1 erhöht werden. 

Ändern Sieden Constructor entsprechend.

 

Klassen-Methoden

Normalerweise sind Methoden immer Instanz-Methoden. An einigen Stellen haben wir aber Instanz-Methoden definiert, die besser als Klassen-Methoden definiert werden. 

In welchen Fällen sollte man anstelle von Instanz-Methoden besser Klassen-Methoden verwenden?

Antwort: Wenn die Methode nicht mit Eigenschaften der Instanz arbeitet. Oder, wenn die Methode aufgerufen werden soll, ohne dass eine Instanz dafür als Ziel-Objekt zur Verfügung steht.

Klassen-Methoden definieren und aufrufen

Solche Methoden, bzw. Stellen im Code sind

  • Das Erzeugen einer Zufallszahl
  • Das Erzeugen einer ZufallsFarbe
  • Das Generieren einer laufenden Nummer für die ID einer Karte
  • Das Erzeugen einer zufälligen Karte, d.h. einer Instanz einer zufällig ausgewählten Subklasse von Karte

Im Folgenden ändern Sie die Definitionen der bisherigen Instanz-Methoden in Definitionen von Klassen-Methoden um. D.h. danach gibt es die Methode nur noch als Klassen-Methode.

Achten Sie auch hier darauf, dass Sie beim Aufruf jeweils den Namen der Klasse angeben. Das ist zwar nicht immer nötig, macht den Code aber klarer, besser verständlich und wartbarer.

Zufallszahl erzeugen

Das Erzeugen einer zufälligen Zahl ist sicher nicht von einem bestimmten Objekt abhängig. Bisher haben wir die entsprechende Methode gleich in zwei Klassen als Instanz-Methode definiert. Beide sollen nun ersetzt werden durch eine einzige Klassen-Methode. 

Ändern Siedie Definition der entsprechende Instanz-Methode so, dass sie nun eine Klassen-Methode der Klasse Karte ist.

Ändern Sie den Aufruf der Methode an allen entsprechenden Stellen.

Die Klassen-Methode soll in allen Klassen des Spiels verwendet werden können.

Zufällige Farbe erzeugen

Das Erzeugen einer zufälligen Farbe ist ebenfalls nicht von einem bestimten Objekt abhängig.

Ändern Sie die Definition der entsprechende Instanz-Methode so, dass sie nun eine Klassen-Methode der Klasse Karte ist.

Ändern Sie den Aufruf der Methode an den entsprechenden Stellen.

 

Zufällige Karte erzeugen

Das Erzeugen einer neuen Karte hat mehr mit der Klasse Karte, als mit der Klasse Spielfeld zu tun.

Ändern Sie die Definition der entsprechende Instanz-Methode der Klasse Spielfeld so, dass sie nun eine Klassen-Methode der Klasse Karte ist.

Ändern Sieden Aufruf der Methode an der entsprechenden Stelle. 

Zur Einordnung: Die Karte ist das "eigentliche" Objekt, das Spielfeld spielt beim Erzeugen einer Karte nur eine Rolle als Parameter. 
Für Fortgeschrittene: Eine solche Methode wird auch als Factory-Methode bezeichnet.

Vorgehensweise, Top-Down und OO

Weiter im Spiel: Nach jedem Klick soll daher folgendener Vorgang ablaufen, den wir in Top-Down Art beschreiben:

  • Wenn mindestens N Nachbarn zueinander passenn, werden sie aus dem Spielfeld genommen
    • Wenn mindestens N Nachbarn zueinander passen, werden sie markiert
      • Mindest-Anzahl N festlegen  
      • Markieren heißt: einen bestimmten Text setzen
      • In Zeilen nach mindestens N passenden Nachbarn suchen und markieren
      • In Spalten nach mindestens N passenden Nachbarn suchen und markieren
    • Als gleich markierte Karten aus dem Spielfeld nehmen. 
      • Für jede Karten aus dem Spielfeld abfragen, ob sie markiert ist
        • Wenn ja, wird sie vom Spielfeld genommen
          • Das bedeutet hier: sie wird durch eine neu erzeugte, zufällige Karte ersetzt
          • Außerdem werden Punkte für diese erfolgreiche Aktion vergeben

Für die Objekt-orientierte Umsetzung wird natürlich außerdem überlegt, welcher Klasse jedes Element zugeordnet wird.

Der komplette Spielverlauf

Der komplette Spielverlauf sieht nun so aus:

Das Ziel des Spiels ist es, mindestens drei zueinander passende Karten nebeneinander zu bekommen, horizontal oder vertikal. Gelingt Ihnen das, bekommen Sie Punkte. Dann werden die zueinander passenden Karten durch neue Karten ersetzt und das Spiel geht weiter. Das Spiel wird beendet, wenn Sie auf den Ende-Button klicken.

Komplettierung des Ablaufs

In der Umsetzung sieht der Ablauf also so aus:

  • Solange der Ende-Button nicht gedrückt wurde
    • Hole einen String von der Konsole
    • Wenn das nicht der Ende-Button ist
      • Ermittle die geklickte Karte aus dem Spielfeld
      • Rufe die Klick-Reaktion der Karte auf 
      • Suche im ganzen Spielfeld nach passenden Nachbarn und markiere sie
      • Nehme alle markierten Karten aus dem Spielfeld
