5Crush nun auch grafisch darstellen

Das Spiel 5Crush soll natürlich auch grafisch dargestellt werden. 

Dafür

  • entwickeln Sie die Klasse Karte Schritt für Schritt weiter und
  • nutzen dabei das neue Feature Grafik-Konsole.

Dabei üben Sie den Umgang mit Instanz-Variablen und Instanz-Methoden.

Klasse Karte weiterentwickeln

In dieser und den folgenden Aufgaben entwickeln Sie die bereits geschriebenen Klasse(n) weiter:

- Sie verwenden dabei die bisher erstellten Instanz-Variablen und Instanz-Methoden weiter. Diese sollten also funktionieren.

- In dieser und den kommenden Aufgaben werden Sie das bereits erstellte Java-File ergänzen und ggf. bereits vorhandene Methoden anpassen und erweitern. 

  • Dazu kopieren Sie daher das File Karte.java, das Sie für die letzte Aufgabe erstellt haben und fügen Sie es in den Ordner dieses Projekts ein. Das machen Sie in Ihrem Betriebssystem, d.h. im Windows Explor oder im Finder. 
    • Benennen Sie es nicht um! Das Java-File muss so heißen, wie die Klasse - und die heißt weiterhin Karte.
  • Um ab jetzt mit diesem File in 5Code arbeiten zu können, schließen Sie dieses Projekt und öffnen Sie es wieder.
    • So ist sichergestellt, dass alle Daten und Verweise im Projekt konsistent sind.
Inzwischen ist der Umfang des Codes in Karte.java schon recht ansehnlich.

Anordnung von Text-Objekten im Raster

Sie hatten bereits eine Methode definiert, die ein Karten-Objekt an eine Position legt, die durch Zeile und Spalte angegeben wird. 

Die neuen grafische Text-Objekte werden für das Spiel auf der Grafik-Konsole an x- und y-Koordinaten dargestellt. Dabei sollen sie in einem festen Pixel-Raster angeordnet werden. Da Texte typischerweise länger als hoch sind, sollen daher für die Zeilenhöhe und die Spaltenbreite unterschiedliche Werte verwendet werden.

Die Zusammenhänge werden von folgender Skizze dargestellt: 

Definieren Sie zwei Instanz-Variablen für diese beiden Rasterangaben (Zeilenhöhe und Spaltenbreite)- natürlich mit einer passenden Javadoc-Beschreibung.

Initialisieren Sie diese beiden Instanz-Variablen direkt bei der Definition mit den in der obigen Skizze angegebenen Werten. D.h. weisen Sie ihnen direkt bei der Definition den entsprechenden Wert zu.

Wenn Sie schon mehr über solche allgemeinen Werte wissen ...

Die Änderung der Position ändert nun auch das Text-Objekt

Die Position der linken oberen Ecke eines grafischen Text-Objekts wird mit x und y in Pixeln angegeben (siehe Beschreibung). 

Ihre Methode zur Positionsänderung einer Karte muss nun wie folgt ergänzt werden:

  • Wenn die Karte bereits ein Text-Objekt referenziert
    • d.h. wenn die IV für das Text-Objekt der Karte nicht null ist
  • Dann bewegen Sie das Text-Objekt an die x/y-Position, die der Angabe Zeile/Spalte entspricht.
    • Dafür verwenden sie die Instanz-Methode moveTo(x,y) der Klasse Text, die im Dokument zur Grafik-Konsole beschrieben ist.
Aus der Angabe Zeile/Spalte im Raster wird eine Pixel-Position in x/y errechnet.

Ergänzen Sie Ihre Methode entsprechend - passen Sie auch die Beschreibung im Javadoc-Format an!

Die Methode moveTo(x,y) ist eine Instanz-Methode der Klasse Text, und nicht der Klasse Karte
Die Prüfung, ob ein Text-Objekt überhaupt referenziert wird ist ratsam ...

Farbe ändern

Das Ändern der Farbe soll nun - sofern die Farbe gültig ist - nicht nur die IV im Karten-Objekt ändern, sondern soll auch die Farbe des grafischen Text-Objekts ändern. 

Dafür ergänzen Sie die bereits geschriebene Methode, in der die Farbe einer Karte geändert wird - sowohl im Code, als auch in der Javadoc-Beschreibung - (Siehe Aufgabe 07 - 2.3.2) wie folgt:

Wenn es ein referenziertes Text-Objekt gibt, ändern Sie dessen Farbe mit dessen (!) Methode setFillcolor(_farbe), die im Dokument zur Grafik-Konsole beschrieben ist.

Position tauschen und Aussehen übernehmen

Die bereits in der letzten Aufgabe definierten Methoden

  • zum Tauschen der Position mit einer anderen Karte
  • zum Übernehmen des Aussehens einer anderen Karte

brauchen wir gar nicht an die neue Situation anzupassen. 

Warum ist eine Anpassung nicht nötig?

Hier zeigt sich ein wichtiger Vorteil des Denkens in Objekt-Kapselung:

Zum Ändern des Zustands eines Objekts werden immer Methoden aufgerufen, diese kümmern sich dann um die notwendigen Schritte in der aktuellen Implementierung des Objekts. 

So kann man sich sowohl Arbeit sparen, als auch Fehler vermeiden.
